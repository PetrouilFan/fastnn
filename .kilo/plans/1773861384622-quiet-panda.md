# Plan: Optimize fastnn ReLU Performance vs PyTorch

## Problem Statement
fastnn's ReLU operation is 24x slower than PyTorch on large tensors (1000×1000). While fastnn is 1.14x faster on small tensors (100×100), the performance degrades significantly with scale. The goal is to close this gap and achieve competitive or superior performance.

## Root Cause Analysis

### 1. Memory Alignment Issues
- **Current**: `Vec<u8>` allocation with `vec![0u8; nbytes]` provides no alignment guarantee
- **Impact**: Forces `_mm256_loadu_ps` (unaligned loads) instead of `_mm256_load_ps` (aligned)
- **Estimated Performance Loss**: 10-15% for memory-bound operations like ReLU

### 2. Insufficient Loop Unrolling
- **Current AVX2**: Processes 2×8=16 elements per iteration
- **PyTorch AVX2**: Processes 4×8=32 elements per iteration
- **Impact**: Higher loop overhead, lower instruction-level parallelism

### 3. Missing Prefetching
- **Current**: No `_mm_prefetch` instructions anywhere
- **PyTorch**: Prefetches 2-3 cache lines ahead to hide memory latency
- **Impact**: ~8-12% performance loss due to memory stalls

### 4. No Streaming Stores
- **Current**: Uses `_mm256_storeu_ps` for all writes
- **PyTorch**: Uses `_mm256_stream_ps` for large arrays (bypass cache)
- **Impact**: 15-20% for arrays > L2 cache size

### 5. Parallelization Overhead
- **Current**: Rayon with fixed 64KB chunk size
- **Issue**: Doesn't adapt to CPU topology or cache hierarchy
- **Impact**: 3-5% overhead from suboptimal chunk sizing

### 6. Small Tensor Inefficiency
- **Current**: Scalar-only path for tensors ≤ 2048 elements
- **Impact**: 20-30% slower for small-medium tensors

### Phase 1: Quick Wins (Low Effort, High Impact)

#### 1.1 Enable Prefetch Feature
**File**: `Cargo.toml`
**Change**: Add `prefetch` to default features
**Impact**: 8-12% for all operations
**Effort**: 5 minutes

#### 1.2 Fix Small Tensor SIMD Path
**File**: `src/kernels/cpu.rs` (lines 4310-4357)
**Change**: Always use SIMD even for small tensors (remove scalar fallback)
**Impact**: 20-30% for tensors ≤ 2048 elements
**Effort**: 1 hour

#### 1.3 Increase AVX2 Loop Unrolling

## Validation Plan

### Benchmark Suite
1. **Microbenchmarks**: Measure individual optimizations in isolation
2. **Integration Tests**: Ensure no regressions in existing functionality
3. **Memory Safety**: Run Miri and Valgrind for memory leak detection
4. **Cross-Platform**: Test on x86_64 (AVX2/AVX512) and aarch64 (NEON)

### Performance Metrics
1. **Latency**: Mean time per operation (μs)
2. **Throughput**: Operations per second (OPS)
3. **Speedup**: Relative to PyTorch CPU implementation
4. **Memory Usage**: Peak memory consumption

## Alternative Approaches Considered

### 1. Using `packed_simd` or `std::simd`
- **Pros**: More portable SIMD code
- **Cons**: Nightly Rust required, potential stability issues
- **Decision**: Rejected due to stability concerns

### 2. Intel MKL Integration
- **Pros**: Potentially superior performance
- **Cons**: Adds heavy dependency, licensing complexity
- **Decision**: Rejected due to dependency size

### 3. Compiler Auto-Vectorization
- **Pros**: Simpler code, compiler-optimized
- **Cons**: Less control over optimization details
- **Decision**: Considered for Phase 4 if manual optimization insufficient
