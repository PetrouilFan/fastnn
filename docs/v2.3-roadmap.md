# v2.3 Roadmap

This document summarizes planned work for fastnn v2.3. Items are prioritized by impact and dependency order.

**Current stable release: v2.2.4**

## P0 — macOS Release-Test Restoration

The release workflow currently skips macOS Rust lib tests due to runner-specific runtime path issues that caused v2.2.2 failures. Restoring macOS test coverage is critical for release confidence on Apple Silicon hardware.

**Scope:**
- Investigate and fix the macOS runtime path issue in cargo test
- Re-enable macOS lib tests in `.github/workflows/release.yml`
- Verify ARM NEON kernels pass on macOS runners (not just Linux cross-compile)

**Status:** Not started. Blocked on GitHub Actions runner environment investigation.

## P1 — WGPU Pooling / Cache Work

Reduce GPU memory pressure and recompilation overhead for repeated inference/training passes.

**Scope:**
- WGPU-side pooling for transient output, staging, scratch, and parameter/state buffers
- Cache specialized matmul and reduction bind groups by operation, dtype, shape class, and layout class
- Shader variant caching to avoid recompilation for repeated op patterns

**Status:** Not started. Depends on profiling data to identify highest-impact cache targets.

## P1 — Optimizer Host-Sync Work

Remove scalar host synchronization inside optimizer loops that currently serialize GPU execution.

**Scope:**
- Eliminate host-device round-trips in Muon normalization and Newton-Schulz steps
- Batch optimizer state reads/writes to reduce synchronization points
- Profile and benchmark improvement from reduced sync overhead

**Status:** Not started. Requires careful numerical validation to ensure optimizer convergence is preserved.

## P2 — Packed Activation / Optimizer State

Explore numerically tolerant paths where activations and optimizer state can use reduced precision.

**Scope:**
- Packed activation storage (INT8/FP16 activations during training)
- Packed optimizer state (FP16 m/v for Adam, reducing memory by 2x)
- Numerical stability analysis for low-bit training paths

**Status:** Not started. This is exploratory research — no guarantees on timeline or scope.

## P2 — ONNX Training Export

Enable autograd through the ONNX DAG and export `.fnn` graphs that preserve gradient flow for interoperability with other frameworks.

**Scope:**
- Export ComputeGraph to ONNX format with backward nodes
- Preserve gradient flow metadata in ONNX graph
- Validate exported models can be loaded and trained in PyTorch/other frameworks

**Status:** Not started. Depends on ONNX training op support maturity in the broader ecosystem.

## Dependency Graph

```
P0 macOS tests  ──── No dependencies, can start immediately
P1 WGPU cache   ──── Needs profiling data
P1 optimizer     ──── Needs profiling data + numerical validation
P2 packed state  ──── Research-first, exploratory
P2 ONNX export   ──── Ecosystem-dependent
```

## How to Contribute

Each item should be implemented on a feature branch (e.g. `agent/wgpu-pooling`), tested, and merged to dev before being included in a release. See `docs/release-process.md` for how to include completed work in a release.
