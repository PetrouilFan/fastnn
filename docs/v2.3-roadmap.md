# v2.3 Roadmap

This document tracks the remaining work for fastnn v2.3 after the post-v2.2.4 stabilization and WGPU performance sprint.

**Current stable release: v2.2.4**

## Completed after v2.2.4

These items were previously listed as planned v2.3 work and are now implemented on `main`:

- **macOS release-test restoration**
  - Fixed macOS aarch64 test instability by zero-initializing CPU arena allocations.
  - Restored macOS runner coverage in `.github/workflows/release.yml`.
- **WGPU pooling and shader/pipeline caching**
  - Added bucketed transient WGPU storage-buffer pooling.
  - Added shader and compute-pipeline cache paths for repeated WGPU dispatches.
  - Added cache telemetry tests to catch hit/miss accounting regressions.
- **Optimizer host-sync reduction**
  - Reduced unnecessary host synchronization in compiled-training optimizer paths.
  - Remaining work is now an audit/benchmarking item rather than a blank implementation item.
- **Release hygiene**
  - Added/updated release preflight checks.
  - Removed stray root-level utility/test files that were not part of the maintained public suite.

## P0 — Roadmap and benchmark truthfulness

Keep public docs aligned with the current implementation before cutting the next release.

**Scope:**
- Keep README, `BENCHMARKS.md`, and this roadmap synchronized with package versions and implemented features.
- Require reproducible benchmark commands for public speed claims.
- Separate maintained CI/regression benchmarks from ad-hoc exploratory benches.

**Status:** In progress. The immediate doc refresh is complete when README and benchmark docs no longer describe stale v2.2.0/v2.3 work.

## P1 — Maintained WGPU benchmark baseline

The CPU `cpu_baselines` suite is maintained and reproducible. WGPU now needs the same level of benchmark hygiene before making public performance claims about pooling and caching.

**Scope:**
- Promote a WGPU benchmark entrypoint to maintained status.
- Measure cold vs warm shader/pipeline latency.
- Track buffer-pool hit/miss or allocation behavior where available.
- Cover at least matmul, fused matmul+activation, and quantized U4/U8 inference paths.
- Document the exact command, hardware context, and output format.

**Status:** Not started. This should be completed before labeling the next release as a performance milestone.

## P1 — WGPU-resident arena execution

WGPU dispatch paths still stage data through CPU memory for several ops. Pooling and caching reduce overhead, but the highest-impact backend improvement is keeping inputs/intermediates resident in GPU buffers across a compiled plan.

**Scope:**
- Represent compiled-plan inputs, intermediates, and outputs as slices of a GPU-resident arena.
- Dispatch chained WGPU ops without per-op CPU extraction and buffer recreation.
- Preserve explicit final readback for user-visible outputs.
- Add tests proving intermediates remain GPU-resident across multi-op graphs.
- Benchmark chained matmul/fusion workloads before and after.

**Status:** Not started. This is the main v2.3 engineering target.

## P1 — Optimizer sync audit and numerical validation

Host synchronization was reduced, but the compiled-training optimizer path still needs a targeted audit before claiming the problem is solved.

**Scope:**
- Benchmark compiled-training steps for SGD, Adam, AdamW, Lion, RMSprop, and Muon.
- Count readbacks/synchronization points per step.
- Add numerical equivalence tests against CPU/reference implementations for small tensors.
- Isolate Muon normalization/Newton-Schulz behavior and verify convergence-sensitive math.

**Status:** Partially complete. Requires benchmark and numerical validation before closing.

## P2 — Fastnn-native training graph export metadata

ONNX training export should not be promised as a direct interoperability feature until the ONNX training ecosystem can faithfully represent fastnn's optimizer/training opcodes. The near-term path is fastnn-native metadata plus clearer ONNX failure modes.

**Scope:**
- Export `.fnn`/`.fnnc` training graphs with backward graph and optimizer-state metadata.
- Keep ONNX export inference-focused unless a faithful training-op mapping exists.
- Ensure training opcodes fail loudly in ONNX export instead of silently dropping gradient flow.
- Document the supported export modes.

**Status:** Not started. Recommended over broad ONNX-training promises.

## P2 — Packed activation / optimizer state research

Explore numerically tolerant reduced-precision paths for training memory reduction.

**Scope:**
- Packed activation storage using INT8/FP16 where numerically safe.
- Packed optimizer state such as FP16 Adam moments.
- Stability analysis and regression tests for low-bit training paths.

**Status:** Research-first. Do not promote to release-blocking work until a small numerical study passes.

## Deferred / future

These remain interesting but should not displace the P1 backend work:

- Multi-GPU training with real device-resident gradient synchronization.
- 2-bit quantization.
- Sparse computation.
- Process-based multiprocessing for Python data loading.

## Dependency graph

```text
P0 docs/bench truthfulness ──> v2.3 release readiness
P1 WGPU benchmarks       ────> public performance claims
P1 WGPU-resident arena   ────> largest backend speedup opportunity
P1 optimizer audit       ────> credible compiled-training performance claims
P2 training export       ────> clearer interop story
P2 packed state          ────> research-first memory reduction
```

## Release recommendation

- Use **v2.2.5** for a stabilization-only release.
- Use **v2.3.0** once WGPU benchmark baselines exist and the docs clearly distinguish completed work from remaining backend work.
